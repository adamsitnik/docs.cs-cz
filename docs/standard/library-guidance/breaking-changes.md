## <a name="types-of-breaking-changes"></a><span data-ttu-id="17a96-122">Typy změny způsobující chyby</span><span class="sxs-lookup"><span data-stu-id="17a96-122">Types of breaking changes</span></span>

<span data-ttu-id="17a96-123">Zásadní změny se dělí do různých kategorií a nejsou rovnoměrně dopad.</span><span class="sxs-lookup"><span data-stu-id="17a96-123">Breaking changes fall into different categories and aren't equally impactful.</span></span>

### <a name="source-breaking-change"></a><span data-ttu-id="17a96-124">Rozbíjející změny zdroje</span><span class="sxs-lookup"><span data-stu-id="17a96-124">Source breaking change</span></span>

<span data-ttu-id="17a96-125">Zdroj narušující změna nemá vliv na provádění programu, ale způsobí chyby při kompilaci, které se je znovu zkompilovat aplikaci.</span><span class="sxs-lookup"><span data-stu-id="17a96-125">A source breaking change doesn't affect program execution but will cause compilation errors the next time the application is recompiled.</span></span> <span data-ttu-id="17a96-126">Například nová přetížení, můžete vytvořit nejednoznačnost volání metod, které byly dříve jednoznačný nebo přejmenováno parametr přeruší volajícím, které používají pojmenované parametry.</span><span class="sxs-lookup"><span data-stu-id="17a96-126">For example, a new overload can create an ambiguity in method calls that were unambiguous previously, or a renamed parameter will break callers that use named parameters.</span></span>

```csharp
public class Task
{
    // Adding a type called Task could conflict with System.Threading.Tasks.Task at compilation
}
```

<span data-ttu-id="17a96-127">Protože zdroji narušující změna je škodlivých pouze když vývojáři znovu zkompilovat svých aplikací, je nejméně rušivé narušující změně.</span><span class="sxs-lookup"><span data-stu-id="17a96-127">Because a source breaking change is only harmful when developers recompile their applications, it's the least disruptive breaking change.</span></span> <span data-ttu-id="17a96-128">Vývojáři můžete snadno vyřešit vlastní přerušeno zdrojový kód.</span><span class="sxs-lookup"><span data-stu-id="17a96-128">Developers can fix their own broken source code easily.</span></span>

### <a name="behavior-breaking-change"></a><span data-ttu-id="17a96-129">Rozbíjející změny chování</span><span class="sxs-lookup"><span data-stu-id="17a96-129">Behavior breaking change</span></span>

<span data-ttu-id="17a96-130">Změny chování jsou nejběžnějším typem rozbíjející změny: téměř všechny změny v chování, které by mohly narušit někdo.</span><span class="sxs-lookup"><span data-stu-id="17a96-130">Behavior changes are the most common type of breaking change: almost any change in behavior could break someone.</span></span> <span data-ttu-id="17a96-131">Změny do knihovny, jako jsou podpisy metod výjimky vyvolána nebo vstupní nebo výstupní formáty dat, může všechny mít negativní vliv na zákazníky knihovny.</span><span class="sxs-lookup"><span data-stu-id="17a96-131">Changes to your library, such as method signatures, exceptions thrown or input or output data formats, could all negatively impact your library consumers.</span></span> <span data-ttu-id="17a96-132">I oprava chyby kvalifikovat jako rozbíjející změny, pokud uživatelé spoléhali na dříve nefunkční chování.</span><span class="sxs-lookup"><span data-stu-id="17a96-132">Even a bug fix can qualify as a breaking change if users relied on the previously broken behavior.</span></span>

<span data-ttu-id="17a96-133">Přidání funkcí a vylepšení chybné chování je dobrá věc, ale bez péče o to může být velmi obtížné pro stávající uživatele k upgradu.</span><span class="sxs-lookup"><span data-stu-id="17a96-133">Adding features and improving bad behaviors is a good thing, but without care it can make it very hard for existing users to upgrade.</span></span> <span data-ttu-id="17a96-134">Jedním z přístupů k pomáhá vývojářům řešit rozbíjející změny v chování je skrýt za nastavení.</span><span class="sxs-lookup"><span data-stu-id="17a96-134">One approach to helping developers deal with behavior breaking changes is to hide them behind settings.</span></span> <span data-ttu-id="17a96-135">Nastavení umožňují vývojářům aktualizovat na nejnovější verzi knihovny, zatímco ve stejnou dobu, zvolíte-li vyjádřit výslovný souhlas nebo vyjádřit výslovný nesouhlas rozbíjející změny.</span><span class="sxs-lookup"><span data-stu-id="17a96-135">Settings let developers update to the latest version of your library while at the same time choosing to opt in or opt out of breaking changes.</span></span> <span data-ttu-id="17a96-136">Tato strategie umožňuje vývojářům dění přičemž umožníte jejich časově náročný kód upravit v čase.</span><span class="sxs-lookup"><span data-stu-id="17a96-136">This strategy lets developers stay up to date while letting their consuming code adapt over time.</span></span>

<span data-ttu-id="17a96-137">Například technologie ASP.NET Core MVC má koncept [verze kompatibility](/aspnet/core/mvc/compatibility-version) aplikací, které mění funkcí aktivovaných a deaktivovaných na `MvcOptions`.</span><span class="sxs-lookup"><span data-stu-id="17a96-137">For example, ASP.NET Core MVC has the concept of a [compatibility version](/aspnet/core/mvc/compatibility-version) that modifies the features enabled and disabled on `MvcOptions`.</span></span>

<span data-ttu-id="17a96-138">**✔️ ZVAŽTE** opuštění nové funkce ve výchozím nastavení, dojde k jejich vliv na stávající uživatele a umožňuje vývojářům vyjádřit výslovný souhlas s funkcí s nastavením.</span><span class="sxs-lookup"><span data-stu-id="17a96-138">**✔️ CONSIDER** leaving new features off by default, if they affect existing users, and let developers opt in to the feature with a setting.</span></span>

### <a name="binary-breaking-change"></a><span data-ttu-id="17a96-139">Binární zásadní změna</span><span class="sxs-lookup"><span data-stu-id="17a96-139">Binary breaking change</span></span>

<span data-ttu-id="17a96-140">Zásadní změna binární soubor se stane, když změníte veřejné rozhraní API knihovny, tak sestavení zkompilována proti starší verze knihovny není nadále možné volat rozhraní API.</span><span class="sxs-lookup"><span data-stu-id="17a96-140">A binary breaking change happens when you change the public API of your library, so assemblies compiled against older versions of your library are no longer able to call the API.</span></span> <span data-ttu-id="17a96-141">Například Změna signatury metody tak, že přidáte nový parametr způsobí, že sestavení, proti starší verzi knihovny vyvolávat <xref:System.MissingMethodException>.</span><span class="sxs-lookup"><span data-stu-id="17a96-141">For example, changing a method's signature by adding a new parameter will cause assemblies compiled against the older version of the library to throw a <xref:System.MissingMethodException>.</span></span>

<span data-ttu-id="17a96-142">Zásadní změna binární soubor může také dojít k narušení **celé sestavení**.</span><span class="sxs-lookup"><span data-stu-id="17a96-142">A binary breaking change can also break an **entire assembly**.</span></span> <span data-ttu-id="17a96-143">Přejmenování sestavení `AssemblyName` změní identitu sestavení, jako se přidání, odebrání nebo změna silný klíč názvů sestavení.</span><span class="sxs-lookup"><span data-stu-id="17a96-143">Renaming an assembly with `AssemblyName` will change the assembly's identity, as will adding, removing, or changing the assembly's strong naming key.</span></span> <span data-ttu-id="17a96-144">Změna identity sestavení budou přerušeny všechny zkompilovaný kód, který se používá.</span><span class="sxs-lookup"><span data-stu-id="17a96-144">A change of an assembly's identity will break all compiled code that uses it.</span></span>

<span data-ttu-id="17a96-145">**❌ NEPODPORUJÍ** změnit název sestavení.</span><span class="sxs-lookup"><span data-stu-id="17a96-145">**❌ DO NOT** change an assembly name.</span></span>

<span data-ttu-id="17a96-146">**❌ NEPODPORUJÍ** přidat, odebrat nebo změnit silného pojmenování klíče.</span><span class="sxs-lookup"><span data-stu-id="17a96-146">**❌ DO NOT** add, remove, or change the strong naming key.</span></span>

<span data-ttu-id="17a96-147">**✔️ ZVAŽTE** místo abstraktní základní třídy rozhraní.</span><span class="sxs-lookup"><span data-stu-id="17a96-147">**✔️ CONSIDER** using abstract base classes instead of interfaces.</span></span>

> <span data-ttu-id="17a96-148">Přidání nic do rozhraní způsobí, že existující typy, které je implementují selhání.</span><span class="sxs-lookup"><span data-stu-id="17a96-148">Adding anything to an interface will cause existing types that implement it to fail.</span></span> <span data-ttu-id="17a96-149">Abstraktní základní třída umožňuje přidat výchozí virtuální implementace.</span><span class="sxs-lookup"><span data-stu-id="17a96-149">An abstract base class allows you to add a default virtual implementation.</span></span>

<span data-ttu-id="17a96-150">**✔️ ZVAŽTE** uvedení <xref:System.ObsoleteAttribute> na typy a členy, které máte v úmyslu odebrat.</span><span class="sxs-lookup"><span data-stu-id="17a96-150">**✔️ CONSIDER** placing the <xref:System.ObsoleteAttribute> on types and members that you intend to remove.</span></span> <span data-ttu-id="17a96-151">Pokyny pro aktualizaci kódu, aby již nebudete používat zastaralé rozhraní API by měl mít atribut.</span><span class="sxs-lookup"><span data-stu-id="17a96-151">The attribute should have instructions for updating code to no longer use the obsolete API.</span></span>

> <span data-ttu-id="17a96-152">Kód, který volá typy a metody s <xref:System.ObsoleteAttribute> zprávou zadaný pro atribut vygeneruje upozornění sestavení.</span><span class="sxs-lookup"><span data-stu-id="17a96-152">Code that calls types and methods with the <xref:System.ObsoleteAttribute> will generate a build warning with the message supplied to the attribute.</span></span> <span data-ttu-id="17a96-153">Uživatelé dávající upozornění, kteří používají zastaralé rozhraní API zařízení surface čas k migraci tak, že je odebraný zastaralá rozhraní API, většina jsou jeho použití.</span><span class="sxs-lookup"><span data-stu-id="17a96-153">The warnings give people who use the obsolete API surface time to migrate so that when the obsolete API is removed, most are no longer using it.</span></span>

```csharp
public class Document
{
    [Obsolete("LoadDocument(string) is obsolete. Use LoadDocument(Uri) instead.")]
    public static Document LoadDocument(string uri)
    {
        return LoadDocument(new Uri(uri));
    }

    public static Document LoadDocument(Uri uri)
    {
        // Load the document
    }
}
```

<span data-ttu-id="17a96-154">**✔️ ZVAŽTE** typy a metody s <xref:System.ObsoleteAttribute> po neomezenou dobu v knihovnách úrovni střední a nízká.</span><span class="sxs-lookup"><span data-stu-id="17a96-154">**✔️ CONSIDER** keeping types and methods with the <xref:System.ObsoleteAttribute> indefinitely in low and middle-level libraries.</span></span>

> <span data-ttu-id="17a96-155">Odebírá se rozhraní API je binární soubor narušující změna.</span><span class="sxs-lookup"><span data-stu-id="17a96-155">Removing APIs is a binary breaking change.</span></span> <span data-ttu-id="17a96-156">Vzhledem k tomu udržování zastaralých typů a metod, pokud jejich údržba je s nízkými náklady a velké množství technického dluhu nepřidává do své knihovny.</span><span class="sxs-lookup"><span data-stu-id="17a96-156">Considering keeping obsolete types and methods if maintaining them is low cost and doesn't add lot of technical debt to your library.</span></span> <span data-ttu-id="17a96-157">Není odebírání typů a metod může pomoci zabránit nejhorším scénáře uvedených výše.</span><span class="sxs-lookup"><span data-stu-id="17a96-157">Not removing types and methods can help avoid the worst-case scenarios mentioned above.</span></span>
